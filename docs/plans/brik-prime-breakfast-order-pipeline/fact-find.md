# BRIK Prime Breakfast Order Pipeline — Fact-Find Brief

## Scope

### Summary
Guest orders placed via the Prime app (complimentary breakfast and evening drinks) are written to Firebase at `preorder/{guestUuid}/{nightKey}` as a pipe-delimited human-readable string (e.g. `"Eggs (Scrambled) | Bacon, Ham, Toast | Americano, Oat Milk, No Sugar | 09:00"`). The bar screen in Reception reads from a completely different path (`barOrders/breakfastPreorders/{monthName}/{day}/{txnId}`) and expects structured items (`{product, count, lineType, price}`). Nothing bridges these two paths. Guest orders are invisible to bar staff.

A secondary consequence is that `ModalPreorderDetails` in the Comp screen expects `nightValue.breakfast` to hold a `txn_...` transaction ID, from which it then reads `barOrders/breakfastPreorders/.../txnId` to show order details. The text string stored by Prime is not a transaction ID and will never produce a match, so the modal falls through to the text-fallback display.

### Goals
1. When a guest submits a breakfast or evening-drink order via the Prime app, a structured record appears at `barOrders/{breakfastPreorders|evDrinkPreorders}/{monthName}/{day}/{txnId}` so that `PreorderButtons` can render an action ticket for bar staff.
2. The `preorder/{guestUuid}/{nightKey}.breakfast` (or `drink1`) field is updated to hold the generated `txnId` so that `ModalPreorderDetails` can look up the placed order.
3. Both sides of the pipe (Prime write, Reception read) use the same item shape: `{product, count, lineType, price, preorderTime, guestFirstName, guestSurname, uuid}`.
4. NightKey alignment between Prime (service-date position derivation) and Comp screen (`Math.floor(today - checkIn) / msPerDay) + 1`) is verified and either confirmed consistent or fixed.
5. Evening drink slot (`drink1`, and optionally `drink2`) follows the same pipeline pattern as breakfast.

### Non-goals
- Changing the wizard UI in Prime.
- Changing the bar staff order-taking flow (PreorderButtons single-click / double-click handling is unchanged).
- Replacing the pipe-delimited string format as the order submission encoding. The backref write changes the stored `breakfast` field to a txnId; the pipe-string format is preserved as the parser input on the server side and is stored in the new `breakfastText` field for Prime guest display.
- Adding pricing to Prime-originated preorders (they are complimentary; price = 0).
- Internationalisation of order item names.

### Constraints & Assumptions
- Constraints:
  - The Prime functions layer runs as Cloudflare Pages Functions with no persistent compute — any bridge write must happen synchronously inside `onRequestPost`.
  - Firebase RTDB is the only persistence layer in scope.
  - The `txnId` must conform to the existing `txn_YYYYMMDDhhmmssfff` format generated by `generateTransactionId()` in Reception, so that `usePlacedPreorder` and `useCompletedOrder` can locate it by ID.
  - The Prime function has access to `CF_FIREBASE_DATABASE_URL` and `CF_FIREBASE_API_KEY` via env bindings; all writes go through `FirebaseRest`.
- Assumptions:
  - The `FirebaseRest.update()` / `FirebaseRest.set()` primitives support fan-out multi-path writes (confirmed: `createPrimeRequestWritePayload` already uses this pattern).
  - Month name and day string formats used in `barOrders/{preorderType}/{monthName}/{day}` match what `getItalyLocalDateParts()` produces in Reception — this must be verified to prevent path mismatches.
  - Evening drinks map to `drink1` only (the `drink2` field exists in the schema but no UI in Prime exposes it). Confirmed in `EvDrinkOrderWizard`.
  - For the Comp screen nightKey alignment: the Comp screen calculates `night${daysSince + 1}` from `checkInDate` (local midnight diff), while Prime derives nightKey from the position of the `serviceDate` entry in the `preorders` record. Both must produce the same key for a given service date — see Risk table.

## Outcome Contract

- **Why:** When a guest uses the app to choose their breakfast, their order is saved but it never makes its way to the staff working the bar. The bar screen shows no record of it. Staff have no way to know what was ordered or even that an order was placed. Fixing this closes the loop so that every guest order placed in the app shows up as a clear ticket for bar staff to act on.
- **Intended Outcome Type:** operational
- **Intended Outcome Statement:** A fact-find is complete that defines the bridge architecture: where it lives, how the order text is converted to structured bar items, and how a transaction reference is stored so the Comp screen can display placed orders.
- **Source:** operator

### Patterns & Conventions Observed

- **Multi-path fan-out writes**: `createPrimeRequestWritePayload` demonstrates writing to multiple RTDB paths atomically via `FirebaseRest.update('/', payload)`. The bridge should use the same pattern to write both the bar order and the txnId back to `preorder/{guestUuid}/{nightKey}.breakfast` in one round-trip. Evidence: `apps/prime/functions/lib/prime-requests.ts`.
- **placedPreorderSchema contract**: bar preorder records at `barOrders/{type}/{monthName}/{day}/{txnId}` must satisfy `{ preorderTime: string, items: SalesOrderItem[] }`. The `PreorderButtons` component also reads `guestFirstName`, `guestSurname`, `uuid` from the same node — these are not in the schema but are expected at runtime.
- **txnId detection in ModalPreorderDetails**: `nightValue.breakfast?.startsWith("txn_")` is the gate. After the bridge, `preorder/{uuid}/{nightKey}.breakfast` must hold the generated txnId string.
- **Month/day path format**: `PreorderButtons` calls `getItalyLocalDateParts(0)` which returns `{ monthName, day }`. The `monthName` is a full English month name (e.g. `"March"`) and `day` is an **un-padded day string** (e.g. `"1"`, `"14"`, `"31"`) — `getItalyLocalDateParts` uses `String(Number(...))` which strips leading zeros. The Prime function must produce exactly matching un-padded day values. Evidence: `apps/reception/src/utils/dateUtils.ts:461`.
- **Time-gating on PreorderButtons**: The component only subscribes to `breakfastPreorders` between 07:00–11:00 and `evDrinkPreorders` between 18:00–21:00. Orders written outside these windows persist in RTDB but are not shown until the next service window — this is the intended behaviour and need not change.

### Data & Contracts

- Types/schemas/events:
  - `PreorderNight` (Prime, internal): `{ night, breakfast, drink1, drink2, serviceDate? }` — `breakfast`/`drink1`/`drink2` currently hold pipe-delimited strings or `"NA"`. After the bridge, `breakfast` holds the txnId and `breakfastText` holds the pipe-delimited string for Prime display.
  - **Parser item granularity**: each pipe segment maps to one `SalesOrderItem`. For the breakfast sides segment (`"Bacon, Ham, Toast"`), each comma-delimited item becomes a separate `SalesOrderItem` with `count: 1, lineType: "bds", price: 0`. This matches the bar staff view where each side is a distinct line item. For evening drinks, the single pipe segment maps to one item (with modifiers stripped from the product name, as they are informational only).
  - `PlacedPreorder` (Reception): `{ preorderTime: string, items: SalesOrderItem[] }` — validated by `placedPreorderSchema`.
  - `SalesOrderItem`: `{ product: string, price?: number, count: number, lineType?: "bds"|"kds" }`.
  - `PreorderButtonData` (Reception UI, runtime-only): `{ transactionId, preorderType, monthName, day, preorderTime, firstName, surname, uuid, items[] }` — requires `guestFirstName`/`guestSurname`/`uuid` fields on the RTDB node that are not part of `placedPreorderSchema`.
  - The bridge record written to RTDB must satisfy both `placedPreorderSchema` (for `usePlacedPreorder` validation) **and** the runtime fields `PreorderButtons` reads.
- Persistence:
  - Write path for bridge: `barOrders/breakfastPreorders/{monthName}/{day}/{txnId}` or `barOrders/evDrinkPreorders/{monthName}/{day}/{txnId}`.
  - Backref write path: `preorder/{guestUuid}/{nightKey}.breakfast` (or `.drink1`) — overwrites the pipe-delimited string with the txnId.
  - The txnId stored in the preorder field enables the Comp screen to locate the barOrder record by path.
- API/contracts:
  - `POST /api/firebase/preorders` (Prime CF function): input `{ service, serviceDate, value, requestChangeException?, token? }`. Response: `{ success, nightKey, order, updated }`. The response shape is unchanged by this feature; the bridge writes are side-effects within the handler.
  - `FirebaseRest.set(path, value)`: single-path write. `FirebaseRest.update('/', payload)`: multi-path atomic write. Both available in Prime functions.
  - NightKey format: `"night1"`, `"night2"`, etc. Consistent between Prime and Reception.

### Dependency & Impact Map

- Upstream dependencies:
  - `validateGuestSessionToken` — must pass for handler to proceed.
  - `firebase.get("preorder/{guestUuid}")` — existing preorder data read to derive nightKey.
  - `hasServiceEntitlement` — entitlement must pass.
  - `findNightKeyForServiceDate` — locates existing nightKey for the given date.
  - `buildBreakfastOrderValue` / `buildEvDrinkOrderValue` — called on the Prime client, not server-side; the server receives the pre-built string. The bridge must parse or re-encode this string into bar items.
- Downstream dependents:
  - `PreorderButtons` (Reception): will start showing tickets once the bridge writes to the expected RTDB path.
  - `ModalPreorderDetails` / `usePlacedPreorder` (Reception): will correctly look up the order once `nightValue.breakfast` holds a `txn_...` ID.
  - `CompScreen.isEligibleForPreorderTonight`: reads `preorder/{occId}/{nightKey}.breakfast` and checks for `!== "NA"`. After the backref write changes the value to a `txnId`, this check still passes (truthy non-"NA" string) — no regression.
  - `CompScreen` plan column: reads `occPre[tonightKey]?.breakfast ?? "NA"` to display the plan badge (e.g. `"cooked"`, `"continental"`). After the backref write, this field holds a raw txnId, which will display as an unlabelled token in the plan badge. This is a visible regression on the Comp screen for bar staff. The plan must address this — either by reading a separate `breakfastPlan` field for the plan display, or by interpreting txnId-prefixed values as `"preordered"`.
  - `MealOrderPage.existingOrders` (Prime client): reads `snapshot.preorders` and displays `night.breakfast` as text. With the dual-field strategy, `night.breakfastText` is displayed instead of `night.breakfast`. The plan must update `existingOrders` in `MealOrderPage` to read from `breakfastText` when present.
  - `usePreorderMutations.savePreorder` (Reception): writes to `preorder/{uuid}/{nightKey}` — can overwrite the txnId backref if a reception operator saves a preorder change. Impact assessment: acceptable; the reception save is authoritative.
- Likely blast radius:
  - Medium. Changes confined to Prime CF function and Prime client display of existing orders. Reception bar screen and Comp screen are consumers only (no code changes needed on the Reception side assuming the written record shape matches the existing contracts).
  - One regression risk: the Prime "existing orders" display showing txnId raw string instead of the order description.

## Questions

### Resolved

- Q: Does the `FirebaseRest.update()` in Prime support multi-path fan-out (writing to multiple paths atomically)?
  - A: Yes. `createPrimeRequestWritePayload` already uses `update('/', { [path1]: val1, [path2]: val2 })` pattern successfully.
  - Evidence: `apps/prime/functions/lib/prime-requests.ts:33-42`

- Q: What shape does `PreorderButtons` expect for a barOrder record?
  - A: At runtime it reads `{ preorderTime, guestFirstName, guestSurname, uuid, items: [{count, lineType, price, product}] }`. The `placedPreorderSchema` only validates `{ preorderTime, items }` — the extra fields must also be written.
  - Evidence: `apps/reception/src/components/bar/orderTaking/preorder/PreorderButtons.tsx:398-429`

- Q: How does `ModalPreorderDetails` decide whether to show a structured order vs. raw text?
  - A: It checks `nightValue.breakfast?.startsWith("txn_")`. If true, calls `usePlacedPreorder` which reads from `barOrders/breakfastPreorders/{month}/{day}/{txnId}`. If false, shows raw text fallback `<p>Breakfast: {fallbackBreakfast}</p>`.
  - Evidence: `apps/reception/src/components/bar/ModalPreorderDetails.tsx:84-103`

- Q: Does the Comp screen's nightKey calculation match Prime's?
  - A: Prime derives nightKey as the insertion-order position among preorder entries. The Comp screen calculates `night${daysSince + 1}` from `checkInDate`. Both should converge on `night1` for check-in day, `night2` for day 2, etc., **provided** the Prime server creates nightKeys sequentially and the serviceDate always advances by one day. Risk: if a guest skips a night (orders for nights 1 and 3 but not 2), the position-based key and the date-diff key may diverge. However, the backref txnId approach means the Comp screen looks up the barOrder directly by txnId — it does not rely on the nightKey for the lookup itself, only for displaying which night the order belongs to.
  - Evidence: `apps/prime/functions/api/firebase/preorders.ts:205`, `apps/reception/src/components/bar/CompScreen.tsx:33-44`

- Q: What is the correct RTDB path month/day format?
  - A: `PreorderButtons` uses `getItalyLocalDateParts(0)` which returns a full English month name (e.g. `"March"`) and an **un-padded** day string (e.g. `"1"`, `"14"`, `"31"`). The implementation is `String(Number(getItalyLocalDateMMDD(italy).slice(2)))` — `Number("03")` → `3` → `"3"`, not `"03"`. The Prime function must produce exactly this un-padded day format. This requires a dedicated helper function in the Prime CF layer with unit tests.
  - Evidence: `apps/reception/src/components/bar/orderTaking/preorder/PreorderButtons.tsx:385-387`; `apps/reception/src/utils/dateUtils.ts:461`

- Q: How should the pipe-delimited order string be converted to structured bar items?
  - A: The bridge must parse the pipe-delimited string into `SalesOrderItem[]` entries. For breakfast, the string is `"Food (style) | sides | drink, modifiers | HH:MM"` (4 segments max). Each pipe segment becomes one or more product items with `count: 1, lineType: "bds", price: 0`. The preorderTime is the last pipe segment. This mapping is deterministic given the `buildBreakfastOrderValue` output format which is well-tested. The parser lives server-side in the Prime function.
  - Evidence: `apps/prime/src/lib/meal-orders/buildOrderValue.ts`

- Q: What is the risk to `CompScreen.isEligibleForPreorderTonight` if `preorder/{uuid}/{nightKey}.breakfast` is overwritten with a txnId?
  - A: Zero regression. The check is `tonight.breakfast !== "NA"`. A txnId string is truthy and not "NA", so the eligible/not-eligible classification is unaffected.
  - Evidence: `apps/reception/src/components/bar/CompScreen.tsx:54-59`

- Q: Should the existing-orders display in the Prime app show the human-readable order summary after the backref write overwrites `breakfast` with a txnId?

## Planning Constraints & Notes

- Must-follow patterns:
  - All Firebase writes from Prime functions use `FirebaseRest` (no Firebase SDK).
  - Multi-path writes use `update('/', payload)` pattern — do not use sequential `set()` calls for atomically-related data.
  - txnId format must be `txn_YYYYMMDDhhmmssfff` (Rome timezone) to match Reception's `generateTransactionId()` format and ensure bar records are locatable.
  - New logic in `onRequestPost` must be gated: only run the bridge write when `success: true` (i.e., after the preorder write succeeds and a new/updated nightKey is known).
  - Month/day RTDB path segments must be produced by a dedicated helper function with unit tests. The day value must be **un-padded** (e.g. `"1"`, `"14"`) — not zero-padded — to match `getItalyLocalDateParts` output exactly.
- Rollout/rollback expectations:
  - No feature flag needed — the change is purely additive from Reception's perspective (it only starts receiving data it currently doesn't receive).
  - Rollback is a CF Pages redeploy removing the bridge write code.
  - If bridge has already written txnId backrefs, rolling back will leave some `preorder/{uuid}/{nightKey}.breakfast` fields holding txnIds — these will display as "Breakfast transaction not found" in the Comp screen modal (existing fallback path) until they are overwritten by a fresh guest order. Acceptable.

  - Log a structured warning (or `console.warn`) when the pipe-string parser cannot extract a valid preorderTime (malformed input guard).
