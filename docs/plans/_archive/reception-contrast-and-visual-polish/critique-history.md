#### Major

**M1 — WCAG approximation caveat not prominent enough**
The fact-find uses OKLCH L as a proxy for WCAG relative luminance. This is an approximation that can be off by ~10-15% for saturated colours. The contrast ratios cited (e.g. 2.44:1 for fg-muted on surface-2) are directionally correct but should be validated by the `tokens:contrast:check` tool before treating any value as definitive.
→ **Resolution**: Added explicit "approximation" qualifier in the token table header and in the validation path. Planning task must include a contrast:check run as a gate. Addressed in-place.

**M2 — `tokens.css` generation dependency not resolved**
The fact-find notes that `tokens.css` is generated by `build-tokens.ts` but does not confirm whether the build script is runnable or what command to use. This could block a build task.
→ **Resolution**: Added an explicit Open Question (Q1) with a recommended approach (update both files manually in-task since the change is deterministic). Addressed in Open Questions section.

#### Minor

**m1 — `/prepare` and `/search` routes marked "not audited" without confirming StatusButton is shared**
Could leave a planner uncertain whether the component fix propagates.
→ **Resolution**: Added explicit note in the Rehearsal Trace row for route coverage: "fix to StatusButton.tsx propagates to all routes automatically." Addressed in-place.

**m2 — CityTaxPaymentButton 4.21:1 described as "acceptable" without citing the applicable WCAG threshold**
The button uses `text-xs font-medium` — normal text size — so 4.5:1 applies, not 3:1. The fact-find correctly flags this as borderline but should state the threshold clearly.
→ **Resolution**: The fact-find already states "WCAG normal-text AA (4.5:1) technically applies" in that section. Threshold is stated. Minor — no further edit needed.
