# Brikette Sitemap Canonicalization Fact-Find Brief

## Scope
### Summary
Audit and correct the sitemap generation contract for `hostel-positano.com` so the generated sitemap includes only final canonical URLs. The immediate defects are legacy `/directions/*` aliases and localized top-level rooms roots like `/en/dorms`, both of which redirect instead of representing final crawl targets.

### Goals
- Identify the exact generator and route-inventory sources that emit redirecting URLs.
- Separate sitemap eligibility from broader canonical app-route inventories so future regressions are harder to introduce.
- Regenerate the checked-in sitemap artifact from the corrected generator.
- Add regression tests/verification so legacy aliases and redirect roots stay out of sitemap output going forward.

### Non-goals
- Removing legacy redirects themselves from runtime.
- Reworking broader localized-route canonicalization outside sitemap emission.
- Expanding `lastmod` coverage in this task.

### Constraints & Assumptions
- Constraints:
  - The sitemap is generated by a build-time script, not a runtime route.
  - Multiple scripts already consume route inventories for different purposes, so changing one shared list in-place risks collateral regressions.
  - The live site still needs legacy redirects such as `/directions/:slug` and top-level rooms aliases.
- Assumptions:
  - Final sitemap eligibility should be stricter than “valid canonical app URL”.
  - Top-level localized rooms roots are non-canonical in every locale because middleware redirects them to the localized booking route.

## Outcome Contract
- **Why:** The live sitemap is leaking redirecting and legacy alias URLs, which wastes crawl budget and weakens canonical trust.
- **Intended Outcome Type:** measurable
- **Intended Outcome Statement:** Sitemap generation emits only final canonical 200 URLs, excludes legacy `/directions/*` aliases and top-level localized rooms roots, and regenerates `apps/brikette/public/sitemap.xml` without those entries.
- **Source:** operator

## Evidence Audit (Current State)
### Entry Points
- `apps/brikette/scripts/generate-public-seo.ts` - build-time generator for `sitemap.xml`, `sitemap_index.xml`, and `robots.txt`.
- `apps/brikette/src/routing/routeInventory.ts` - source of localized URL inventories consumed by SEO/link tooling.
- `apps/brikette/src/middleware.ts` - runtime redirect contract proving some emitted URLs are not final canonicals.
- `apps/brikette/public/sitemap.xml` - checked-in generated artifact currently containing the bad entries.
- `apps/brikette/scripts/verify-sitemap-contract.ts` - post-generation verifier currently encoding the bad expectation set.

### Key Modules / Files
- `apps/brikette/scripts/generate-public-seo.ts`
  - `listDirectionPaths()` emits `/directions/:slug` aliases directly into the sitemap source list.
  - `listCanonicalSitemapPaths()` currently returns `"/", ...listDirectionPaths(), ...listLocalizedPublicUrls()`.
- `apps/brikette/src/routing/routeInventory.ts`
  - `listLocalizedPublicUrls()` includes every localized top-level section from `PUBLIC_INDEXABLE_SECTION_KEYS`, including `rooms`.
  - The function is used as the sitemap source of truth today, but its current contract is broader than “final sitemap URL”.
- `apps/brikette/src/middleware.ts`
  - `buildLocalizedRoomsRedirectPath()` maps top-level localized rooms roots to the localized booking slug.
  - `maybeRedirectLocalizedTopLevel()` permanently redirects any localized `rooms` top-level path when `nextParts.length === 2`.
- `apps/brikette/public/sitemap.xml`
  - Contains 24 `/directions/*` entries and `/en/dorms` plus localized room-detail URLs.
- `apps/brikette/scripts/verify-sitemap-contract.ts`
  - Treats `listCanonicalSitemapPaths()` as the expected sitemap set, so it currently locks the bad behavior in place.
- `apps/brikette/src/test/lib/generate-public-seo.lastmod.test.ts`
  - Asserts that every `listLocalizedPublicUrls()` path appears in the sitemap, which currently forces `/en/dorms`-style roots into the emitted XML.

### Patterns & Conventions Observed
- Generator scripts prefer deterministic inventories derived from source data and route helpers rather than crawling built output.
  - Evidence: `apps/brikette/scripts/generate-public-seo.ts`, `apps/brikette/scripts/lib/static-runtime-redirects.ts`
- The codebase already distinguishes multiple URL contracts:
  - “canonical app URLs” for internal links/verification.
  - legacy redirect inventories for runtime redirects.
  - sitemap-specific filtering via `shouldExcludeFromSitemap`, though that filter is currently too shallow.
- Legacy aliases are intentionally kept live via redirect rules rather than removed.
  - Evidence: `apps/brikette/scripts/lib/static-runtime-redirects.ts`, `apps/brikette/src/routing/staticExportRedirects.ts`

### Data & Contracts
- Types/schemas/events:
  - `StaticRedirectRule` in `apps/brikette/src/routing/staticExportRedirects.ts` defines runtime redirect and rewrite contracts.
- Persistence:
  - `apps/brikette/public/sitemap.xml` and `apps/brikette/public/sitemap_index.xml` are generated artifacts.
- API/contracts:
  - `middleware.ts` defines a 301 from localized top-level rooms roots to localized booking roots.
  - `_redirects` generation maps `/directions/:slug` to `/en/how-to-get-here/:slug`.

## Questions
### Resolved
- Q: Why are `/directions/*` URLs appearing in the sitemap?
  - A: `generate-public-seo.ts` explicitly prepends `listDirectionPaths()` into `listCanonicalSitemapPaths()`.
  - Evidence: `apps/brikette/scripts/generate-public-seo.ts`
- Q: Is `/en/dorms` a final canonical page or a redirect source?
  - A: It is a redirect source. Middleware permanently redirects any localized top-level rooms route to the localized booking route.
  - Evidence: `apps/brikette/src/middleware.ts`
- Q: Can we remove `/directions/*` from the sitemap without removing runtime support?
  - A: Yes. Redirect support is maintained by `_redirects` and static runtime redirect rules, which are separate from sitemap generation.
  - Evidence: `apps/brikette/scripts/lib/static-runtime-redirects.ts`, `apps/brikette/src/test/migration/url-inventory.test.ts`
- Q: Should the fix overload `listLocalizedPublicUrls()` or introduce a narrower sitemap contract?
  - A: Introduce a narrower sitemap contract. Other tooling already consumes broader route inventories, so sitemap eligibility should be isolated.
  - Evidence: `apps/brikette/scripts/lib/static-runtime-redirects.ts`, `apps/brikette/scripts/verify-rendered-link-canonicals.ts`
