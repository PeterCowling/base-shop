/lp-do-ideas

Ideas

Monitor strategic documents for meaningful changes and queue the Fact-Find work they imply.
What you tell it
  • Artifact ID ‚Äî the identifier of the strategic document to evaluate Required
  • Business ‚Äî which business or product context this relates to Required
  • After SHA ‚Äî a fingerprint of the document's current version Required
  • Path ‚Äî the file path of the document Required
What it does
  • Compares the current document version against the last known version to detect meaningful changes
  • Assesses every detected change using judgment ‚Äî asks: is it material? does it open a planning gap? is the area already covered by in-flight work? Routes to Fact-Find if investigation is needed (any domain: strategy, offer, distribution, product, measurement, brand, supply), Briefing if understanding-only, or no-action if the change is minor or already handled
  • No changed sections, or a first-time registration, results in a logged no-action ‚Äî nothing queued
  • Queues the dispatch for your review ‚Äî never auto-invokes follow-up skills without your confirmation
  • Suppresses duplicates ‚Äî the same document-version pair is never queued twice
What it produces
  • A dispatch record queued at ideas/trial/queue-state.json
  • A telemetry log entry ‚Äî what was detected, what route was recommended, and why
  • A suggested next invocation ‚Äî the area anchor, location anchor, and deliverable family pre-filled so you can run /lp-do-fact-find (or /lp-do-briefing) immediately without working out the inputs yourself
Good to know
  • Operates in trial mode ‚Äî it proposes actions, never takes them automatically. You confirm each dispatch
  • Only strategically significant changes trigger a dispatch. Cosmetic or minor edits are logged but produce no action
  • Runs independently of individual feature work ‚Äî it monitors background strategic context
‚Üí After reviewing a dispatch, confirm it ‚Äî then run Fact-Find with the recommended inputs to begin scoping the work.

/lp-do-fact-find

Fact-Find

Scope a feature idea into a structured, evidence-backed discovery document ready for planning.
What you tell it
  • Area anchor ‚Äî the broad domain of the feature (e.g. "checkout", "marketing", "product catalogue") Required
  • Location anchor ‚Äî which product, app, or codebase area is affected Required
  • Provisional deliverable family ‚Äî the type of output expected (e.g. "code feature", "business doc", "content") Required
  • Any existing briefs, research notes, or context documents Optional
What it does
  • Runs a structured discovery: sufficiency gate, work-type classification, module-specific investigation, evidence gap review
  • If it doesn't have enough information to proceed, it pauses and asks targeted questions before continuing
  • Classifies work as code, business doc, or mixed ‚Äî this determines which planning and build modules are used downstream
  • For business-track work: maps the blast radius ‚Äî every other channel, copy asset, UX flow, automation, or policy doc that would become inconsistent if this deliverable ships without them also changing; flags which are in-scope vs out-of-scope with a reason
  • Runs an evidence gap review and assigns a status: Ready-for-planning, Needs-input (stops for more info), or Infeasible (pipeline ends here)
  • Checks a minimum evidence floor before running Critique ‚Äî code track needs at least one entry point and key module; business track needs at least one hypothesis and Delivery-Readiness ‚â•60%; either failure forces Needs-input immediately
  • Automatically runs Critique ‚Äî auto-continues to Plan only when the critique scores ‚â•4.0 (credible); a partially credible result (3.0‚Äì3.5) stops the pipeline and surfaces the findings to address
What it produces
  • A fact-find at docs/plans/<feature-slug>/fact-find.md with status: Ready-for-planning, Needs-input, or Infeasible
  • Sections: scope, evidence audit, open questions, confidence inputs, risks, planning readiness
  • A routing header in frontmatter configuring work type, deliverable family, and execution skill
  • A critique score in critique-history.md appended automatically
Good to know
  • This is the most important step ‚Äî a thorough fact-find leads to a confident, well-scoped plan
  • If it asks you questions, answer them fully ‚Äî partial answers produce weaker downstream plans
  • Critique must score ‚â•4.0 to proceed ‚Äî a 3.0‚Äì3.5 result stops here and tells you exactly what to address before re-running
  • An Infeasible result means the work can't be done as scoped ‚Äî the pipeline ends and you get a kill rationale explaining why
‚Üí When status is Ready-for-planning and Critique scores ‚â•4.0, Fact-Find automatically continues to Plan ‚Äî you don't need to invoke it separately. A score of 3.0‚Äì3.5 stops here first.

/lp-do-plan

Plan

Turn a fact-find into a sequenced, confidence-gated task plan ready for the build skill to execute.
What you tell it
  • The feature slug (e.g. checkout-improvements) ‚Äî it finds the fact-find automatically Required
  • Or a direct path to the fact-find.md document Required
  • Any additional constraints or scope guidance Optional
What it does
  • Reads the fact-find and decomposes the work into discrete, ordered tasks
  • Assigns each task a confidence score (0‚Äì100%). Tasks below 80% without an upstream investigation task to cover them trigger an automatic Critique
  • Sets up dependency chains and runs /lp-do-sequence to topologically sort tasks, detect parallelism, and produce a Parallelism Guide so Build can dispatch wave tasks concurrently
  • Defines validation contracts per task ‚Äî specifying what "done" looks like before a task counts as complete
  • Runs Critique automatically when overall confidence is low or any task is uncovered; in pipeline mode, auto-continues to Build without waiting
What it produces
  • A plan at docs/plans/<feature-slug>/plan.md ‚Äî status Draft until build is authorised, then Active
  • Ordered task list with: type, confidence score, dependencies, files affected, and validation contracts
  • A parallelism guide showing which tasks can run simultaneously
Good to know
  • The plan is a living document ‚Äî it can be updated at any time using Replan
  • When invoked from the pipeline (after Fact-Find), Plan will auto-continue to Build if all gates pass ‚Äî no manual step needed
  • The plan is a plain Markdown file ‚Äî you can read and edit it at any time
‚Üí With a plan in place, run Build to execute tasks one at a time. Each call executes one task, updates the plan, and stops.

/lp-do-build

Build

Execute one plan task at a time ‚Äî safely gated, validated, and committed with full evidence.
What you tell it
  • The feature slug or plan file path ‚Äî it finds the next runnable task automatically Required
  • A specific task ID if you want to target a particular task Optional
  • The fact-find brief for additional context Optional
What it does
  • Selects the next eligible task from the plan (respects dependencies, confidence thresholds, and wave order)
  • Checks 5 safety gates before touching any file: eligibility, scope, validation requirements, commit safety, and post-task updates
  • Executes the task ‚Äî writes code, creates documentation, or performs research depending on task type
  • Validates the output matches the task's defined contract before marking it done
  • If a task is a DECISION (requires a call you need to make), it stops, writes the question to replan-notes, and waits ‚Äî it never silently skips
  • If a task confidence is too low and no other task in the current wave can raise it, automatically calls Replan rather than guessing
What it produces
  • The built artifact ‚Äî code, documentation, or analysis as specified by the task
  • Plan updated: task marked Complete with date and build evidence
  • A build-record.user.md when all tasks complete ‚Äî immediately followed by /ops-ship to push and watch CI green
  • Git commit(s) with scoped, traceable changes per task
Good to know
  • Build always does one task and stops ‚Äî review after each task before continuing
  • Independent tasks that can safely run in parallel are dispatched together as a wave
  • Shipping happens immediately when all tasks are done. A results review is required separately before the plan is archived ‚Äî but it doesn't block the ship
‚Üî Keep running Build until all tasks are complete. If a task gets blocked or requirements change mid-way, run Replan first then continue.

/lp-do-replan

Replan

Update a plan when things change ‚Äî without losing what's already been built.
What you tell it
  • The feature slug or plan file path Required
  • The task ID(s) that are blocked or uncertain Required
  • What changed ‚Äî new constraints, failed assumptions, new information, scope shifts Optional but helpful
What it does
  • Reads the current plan and identifies what needs changing
  • For low-confidence tasks, creates formal investigation or spike tasks as precursors ‚Äî so Build never proceeds on guesswork
  • Preserves all completed tasks and their evidence exactly as-is
  • Updates task descriptions, dependencies, affected files, and validation contracts as needed
  • If a task has been below threshold for 3+ replan rounds with no viable path forward, declares it Infeasible ‚Äî the pipeline ends with a kill rationale
What it produces
  • An updated plan.md ‚Äî same file, stable task IDs, with revisions applied and re-sequenced
  • A replan-notes.md with a replan round counter and detailed reasoning when changes are significant
  • New investigation/spike tasks pre-inserted ahead of the task that needed their findings ‚Äî or an Infeasible declaration if no viable path exists
‚Üí After replan, return to Build to continue. Build picks up automatically from the newly updated task list.

Critique

A scored quality review of any plan or document ‚Äî with automatic fixes applied where possible.
What you tell it
  • The document to review ‚Äî plan, fact-find, or business artifact Required
  • The schema mode ‚Äî what type of document this is (auto-detected if not specified) Optional
  • Whether to run critique-only or critique + autofix (default: autofix on) Optional
What it does
  • Works on fact-finds, plans, offer artifacts, and process/skill documents ‚Äî auto-detects which lens to apply
  • Scores across 6 weighted dimensions: Evidence quality (25%), Coherence (20%), Completeness (15%), Feasibility (15%), Risk handling (15%), Measurability (10%)
  • Issues are tiered: Critical (blocks a safe decision), Major (meaningfully raises failure risk), Moderate (weakens confidence), Minor (style/template drift)
  • In default autofix mode: applies a 4-step fix process ‚Äî section rewrite gate, point fixes, consistency scan, then issues ledger update
  • Escalates Critical issues it cannot fix on its own; blocks auto-build if critique score ‚â§ 2.5
What it produces
  • Critique score and issue list appended to (or replacing) the document's critique section
  • In-place edits to the document when autofix is enabled
  • An entry in docs/plans/<slug>/critique-history.md ‚Äî a running quality ledger across all passes
Good to know
  • Usually runs automatically ‚Äî triggered by Plan when overall confidence is below 4.0/5.0, or when any task is low-confidence without an upstream investigation covering it
  • Invoke manually for a fresh check on a document you've edited by hand, or on an offer artifact or skill doc
  • Score ‚â§ 2.5 blocks auto-build; score 3.0‚Äì3.5 allows build but adds a warning flag; score ‚â• 4.0 proceeds normally
‚Ñπ Critique is primarily an automatic quality gate. Its output appears at the end of Fact-Find and Plan runs. Invoke it manually for a fresh check on an updated document.

Briefing

Trace how something currently works ‚Äî end-to-end with file and line evidence. Not part of the delivery sequence ‚Äî use it when you need to understand a system before deciding whether to change it.
What you tell it
  • Topic or system area ‚Äî what you want to understand (e.g. "how checkout works", "the email sending pipeline") Required
  • Location anchor ‚Äî a file path, route, endpoint, error/log reference, or user flow to start the investigation from Required
  • Specific questions you want answered Optional
What it does
  • Traces current behavior end-to-end through code, configuration, and data flows ‚Äî with file and line pointers for every non-trivial claim
  • Investigates using read-only commands ‚Äî searches, git history, targeted type checks ‚Äî never changes anything
  • Deliberately does not produce a plan or recommend actions ‚Äî this is a factual snapshot, not direction
What it produces
  • A briefing note at docs/briefs/<topic-slug>-briefing.md
  • Evidence-backed explanation of current behavior ‚Äî what exists, how it connects, where the boundaries are
  • No tasks, no plan, no recommendations ‚Äî a factual snapshot you can use to decide what (if anything) to do next
‚Ñπ Briefing has no handoff ‚Äî it produces a factual snapshot, not a plan. Use /lp-do-fact-find when you are ready to plan a change to what you've just understood.

How the Skills Connect

The core delivery sequence

Ideas ‚Üí Fact-Find ‚Üí Plan ‚Üí Build is the primary delivery sequence. Each skill passes structured output to the next as a file ‚Äî you always have a readable record of what was decided and why.

Build and Replan iterate

Build and Replan work as a pair. Build executes one task and stops. If anything is uncertain, Replan updates the plan. Build resumes. This continues until all tasks are complete.

Critique is automatic

You don't normally invoke Critique directly. It runs automatically after Fact-Find and after Plan to catch quality issues early ‚Äî before Build starts executing.

Briefing is standalone

Briefing is not part of the delivery sequence. Use it any time you want background understanding on a topic. It produces no plan, no tasks, and has no handoff into the sequence.

Where everything lives:  All skill outputs are saved as Markdown files ‚Äî fact-find, plan, build record, and critique history all go under docs/plans/<feature-slug>/, briefing notes under docs/briefs/. Nothing produced by a skill exists only in chat ‚Äî everything is written to file.
