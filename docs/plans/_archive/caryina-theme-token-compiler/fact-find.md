# Caryina Theme Token Compiler Retrofit — Fact-Find Brief

## Scope

### Summary

Brikette has been migrated to a three-layer generated theme token system: assets → design-profile → ThemeCSSConfig bridge → `generateThemeCSS()` compiler → committed generated CSS file. The hand-authored `:root` and `.dark` blocks in `global.css` were replaced with a build artifact, proven by parity tests.

Caryina needs the same retrofit: write `theme-css-config.ts`, generate `apps/caryina/src/styles/theme-tokens.generated.css`, add the two parity test files, and update `global.css` to import the generated file instead of hand-authoring the token blocks.

The scope is materially simpler than brikette: caryina has no brand-scoped `--color-brand-*` namespace (tokens use semantic names like `--color-primary`, `--color-accent`), no RGB triplet vars, no gradients, and empty recipes. The compiler needs a leaner config, and the tests need to target a different dark-mode pattern.

### Goals

- Write `packages/themes/caryina/src/theme-css-config.ts`
- Produce `apps/caryina/src/styles/theme-tokens.generated.css` (committed, replaces the inline token blocks)
- Write `packages/themes/caryina/__tests__/generated-parity.test.ts` mirroring the brikette pattern
- Write `packages/themes/caryina/__tests__/coverage-parity.test.ts` mirroring the brikette pattern
- Update `apps/caryina/src/styles/global.css` to import the generated file instead of the tokens inline
- Export `theme-css-config` from `packages/themes/caryina/package.json`

### Non-goals

- Changes to `packages/themes/base/src/generateThemeCSS.ts` — the compiler is shared and must not be modified for caryina-specific behavior
- Changes to brikette
- Migrating caryina to class-based dark mode (`.dark`) — see Dark Mode Risk section; caryina currently uses `@media (prefers-color-scheme: dark)` + `html.theme-dark`; the generated file should match this
- Adding recipes — `recipes.ts` is empty and correct as-is

### Constraints & Assumptions

- Constraints:
  - The `generateThemeCSS()` compiler emits `:root {}` and `.dark {}` blocks, not `@media` blocks. The generated output format is fixed. This has a meaningful implication for caryina (see Dark Mode Risk section).
  - No net-new tokens may be introduced — the generated output must round-trip exactly to what is currently committed.
  - Tests must run under the existing jest config for the `packages/themes/caryina` package, which does not yet exist (a `jest.config.cjs` and test infrastructure must be added, following the brikette pattern).
- Assumptions:
  - The plan will generate a `.dark {}` class-based output file, and `global.css` will be updated to add `html.theme-dark` as a selector alias (or the import strategy will use the generated file for the `:root` block only, and the dark mode overrides will remain in a separate pattern — see Open Questions).
  - `packages/themes/caryina` does not yet have a jest test infrastructure; it needs to be wired up following `packages/themes/brikette/jest.config.cjs` as reference.

Dark mode is a media/class alias pattern: the dark values are stored as `--color-*-dark` vars in `:root`, then reassigned in `@media (prefers-color-scheme: dark) :root {}` and `html.theme-dark {}`. There are **no standalone dark-variant properties** — just reassignments of the semantic vars.

`global.css` `:root {}` adds two derived vars on top:
- `--brand-mark-color: hsl(var(--color-primary))` — not in tokens.css
- `--brand-accent-color: hsl(var(--color-accent))` — not in tokens.css

### Dark Mode Architecture Difference from Brikette

Brikette uses:
- `packages/themes/brikette/src/theme-css-config.ts` → generates `:root {}` (light) + `.dark {}` (class-based overrides)
- `apps/brikette/src/styles/global.css` uses `.dark` class on `<html>`

Caryina uses:
- `@media (prefers-color-scheme: dark) :root {}` — OS-driven
- `html.theme-dark {}` — explicit class override
- `@media (prefers-color-scheme: dark) :root:not(.theme-dark) {}` in `global.css` — forces light when user picked light despite OS dark

The compiler only emits `.dark {}`, not `@media` blocks. This is a structural mismatch.

**Resolution strategy (see Open Questions):** The generated file should store only the canonical token values — i.e. the `:root` (light) block and a `.theme-dark` block (using the selector `html.theme-dark`). The `@media (prefers-color-scheme: dark)` block remains hand-authored in `global.css` as a cascade alias (or the generated file uses both selectors in one rule). The parity tests compare only `:root` and `html.theme-dark` (or `.dark` — the selector used in the generated file).

Alternatively, the generated file could commit only the `:root` light block, and the dark aliases remain hand-authored. This would be a narrower scope.

### Assets Gap Analysis

**Assets that map directly to tokens:**

| Asset key | Token(s) it feeds |
|---|---|
| `fonts.heading.family` (Cormorant Garamond) | `--font-heading` |
| `fonts.body.family` (DM Sans) | `--font-sans` |
| `brandColors.strawberryMilk` | `--color-primary` (and derivatives: `--color-primary-fg`, `--color-primary-soft`, `--color-primary-hover`, `--color-primary-active`) |
| `brandColors.warmSage` | `--color-accent` (and derivatives: `--color-accent-fg`, `--color-accent-soft`) |
| `brandColors.warmIvory` | `--color-bg` |

**Asset gap:** `assets.brandColors` has only 3 entries (strawberryMilk, warmSage, warmIvory). But `tokens.css` has 15 color vars + 3 radius vars + 2 font vars = 20 vars. The 12 derived color vars (`--color-primary-fg`, `--color-primary-soft`, `--color-primary-hover`, `--color-primary-active`, `--color-accent-fg`, `--color-accent-soft`, `--color-border`, `--color-border-muted`, `--color-border-strong`, `--color-fg`, `--color-fg-muted`, `--color-surface`) are **not** in `assets.brandColors`. They are not derivable from the 3 brand colors via a formula — they are independently defined HSL triplets.

**Resolution:** All 15 color vars (including the derived palette) must be added to `assets.brandColors` for the compiler to emit them. The compiler iterates `colorVarMap` and looks each key up in `assets.brandColors`. Any key absent from `brandColors` is silently skipped. The `assets.ts` file must therefore be extended to include the full palette.

### Profile Gap Analysis

The compiler currently emits exactly one profile var: `--theme-transition-duration: ${profile.motion.durationNormal}`. Caryina's profile has `durationNormal: "250ms"`. This var is **not** present in `tokens.css` (it's not in the current token set). Either:
1. The compiler emits it and the generated file includes it (no corresponding entry in existing `tokens.css` — would be a new token), or
2. The parity test compares only what's in `tokens.css`, which doesn't have `--theme-transition-duration`, so the "no extra vars" inverse check would fail.

**Resolution:** Suppress the profile var emission for caryina, or add `--theme-transition-duration` to `tokens.css` as a new token. Simpler approach: update `generateThemeCSS()` to only emit the profile var when explicitly opted in, or treat it as a `derivedVars.light` entry in the config. Since the compiler is shared and cannot be changed for caryina specifically, the cleanest path is to include `--theme-transition-duration: 250ms` as a `derivedVars.light` entry so it appears in the output, and also add it to the committed `tokens.css` as the new generated content. This is a net-new token added in the migration.

Actually — the simpler read is: the committed `tokens.css` will be **replaced** by the new generated file. The generated file will contain whatever `generateThemeCSS()` emits. So `--theme-transition-duration` will appear in the new generated file, and the parity test compares generated output to generated output (not to the old `tokens.css`). The flow is:

1. Write `theme-css-config.ts`
2. Run compiler to produce `theme-tokens.generated.css`
3. Commit it
4. Tests compare compiler output to committed file — always equal unless someone hand-edits the committed file

So the profile var emission is not a problem: it simply becomes part of the new generated token set. The only concern is that `global.css` currently imports the old `tokens.css` — the import path must be updated to the new generated file.

### Dark Mode Selector Decision

The compiler emits `.dark {}` class selector. Caryina uses `html.theme-dark {}`. Options:

**Option A — Use `.dark` selector in generated file, add `html.theme-dark` alias in `global.css`:**
- Generated file: `:root {}` + `.dark {}`
- `global.css`: add `html.theme-dark { @apply ... }` or a CSS rule that forwards `.dark` vars to `html.theme-dark` context
- Simplest alignment with the compiler's output format
- Keeps parity tests trivial (same selector names as brikette)

**Option B — Keep `html.theme-dark` and `@media` hand-authored, generate only `:root`:**
- Generated file: `:root {}` only (skip `.dark` block)
- Dark mode stays hand-authored
- Narrower scope, no selector mismatch
- Parity test only checks `:root` block
- Does not eliminate the hand-authoring risk for dark values

**Option C — Modify `global.css` to switch to `.dark` class-based dark mode:**
- Align caryina with brikette's dark mode pattern completely
- Requires changing the dark mode toggle in the React app (if it sets `html.theme-dark`, change to `html.dark`)
- Broadest scope; highest risk; out of scope for this ticket

**Recommended: Option A.** The `global.css` currently already has `html.theme-dark {}` — this can be extended to also match `.dark` by changing the selector to `html.theme-dark, html.dark {}` or by adding `.dark {}` as an alias rule. The `@media (prefers-color-scheme: dark) :root {}` block in `tokens.css` becomes part of the generated file as `derivedVars` or a separate committed section.

Actually, the cleanest approach is: the generated file emits `:root {}` and `.dark {}`. The `global.css` adds:
```css
html.theme-dark { @extend .dark; }  /* or simply: */
html.theme-dark, .dark { ... }
```
No — this doesn't work for CSS variables. The simplest clean path: change the dark mode class used on `<html>` from `theme-dark` to `dark` in the React app, aligning with brikette. This needs a search across the caryina app for `theme-dark` usages.

This is an in-scope decision the plan must make explicit.
