# BASE Theme OKLCH Migration Plan

## Summary

Migrate all color tokens in `packages/themes/reception/src/tokens.ts` from HSL to OKLCH
format, sequenced as two separate commits: shade tokens first, semantic tokens second.
A programmatic conversion script handles all value derivations — no manual conversion.
Contrast ratios are verified post-migration using CSS Color 4 math. The `@theme inline`
block in reception's globals.css continues working because shade token values are
referenced via bare `var()`, so changing the underlying value format is transparent.

## Active tasks
- [x] TASK-01: Write HSL→OKLCH conversion script
- [ ] TASK-02: Migrate reception shade tokens to OKLCH (commit 1)
- [ ] TASK-03: Migrate reception semantic tokens to OKLCH + update @theme consumers (commit 2)
- [ ] TASK-04: Contrast ratio verification

## Goals
- All `packages/themes/reception/src/tokens.ts` color values in OKLCH format
- `packages/themes/reception/tokens.css` regenerated with OKLCH values
- `apps/reception/src/app/globals.css` @theme updated for semantic token consumption
- Contrast ratios documented for all migrated tokens

## Non-goals
- Base theme semantic token migration (base/tokens.css raw triplet defaults are only
  fallbacks and are overridden by reception values — base is out of scope for this plan)
- brikette, prime, xa-b theme migration (separate plans if warranted)
- Changing any UI component code

## Constraints & Assumptions
- Constraints:
  - Write conversion script — no hand-computed values
  - Shade tokens and semantic tokens must be committed separately
  - @theme inline in globals.css must continue to work post-migration
  - tailwind-merge@^2.5.5 compat: class names unchanged, OKLCH values don't affect merging
- Assumptions:
  - OKLCH values generated by HSL→OKLCH script are accurate to CSS Color 4 spec
  - @theme inline bare `var()` pattern is format-agnostic (works with any valid color value)
  - `hsl(var(--color-X))` wrappers in globals.css @theme {} and direct CSS will be
    replaced with `var(--color-X)` for semantic tokens

## Inherited Outcome Contract

- **Why:** shadcn/ui v4 uses OKLCH as canonical; existing HSL system has produced
  production cascade bugs; OKLCH improves multi-brand color system quality.
- **Intended Outcome Type:** operational
- **Intended Outcome Statement:** All reception theme token values converted to OKLCH
  with contrast ratios verified, @theme inline continues working, no visual regressions.
- **Source:** operator

## Fact-Find Reference
- Related brief: dispatch IDEA-DISPATCH-20260306213000-1005 + context from IDEA-1004
- Key findings used:
  - @theme inline now live in reception (commit f463fe63b5)
  - Shade tokens already store full hsl() values → @theme inline uses bare var() → format-agnostic
  - Semantic tokens use raw HSL triplets consumed via hsl(var(...)) in @theme {} → requires
    updating @theme entries to var() when migrating to pre-wrapped oklch() values
  - Validated migration pattern: pre-wrap at source, reference via var()

## Proposed Approach
- Option A: Manual OKLCH conversion (error-prone, slow)
- Option B: Programmatic script from existing HSL values → run once, hardcode output
- Chosen approach: Option B — script derives all values, output is hardcoded into source.
  This is auditable and reproducible.
