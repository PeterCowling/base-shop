Startup Loop: File Types Explained

Seven file types power the entire startup loop. Each has a specific job. Understanding what each extension does tells you at a glance what a file is for, who's responsible for it, and whether you need to act.

Documents Needs your input Rendered for reading Schema / config Code Scripts Data
.md
Markdown
The workhorse

The primary format for everything the startup loop reads and writes. Plans, fact-finds, strategy briefs, skill instructions, results reviews ‚Äî they're all .md. Markdown lets the agent read and write structured documents that are also human-readable without any tooling. Over 1,200 of them in the loop alone.

Examples
plan.md fact-find.md HBAG-offer.md SKILL.md market-pack.md

.user.md
Markdown ¬∑ human required
Needs your hand

Same format as .md ‚Äî the .user. part is a naming convention, not a different file type. It signals that human involvement is required before this file is complete: either you wrote the whole thing, or the agent drafted it and you must confirm it. The loop will not advance past a gate until the required .user.md file exists.

Examples
results-review.user.md build-record.user.md brand-dossier.user.md intake-packet.user.md 2026-02-24-lp-forecast.user.md

‚úçÔ∏è You write or confirm. Agent may draft.

.html
Rendered document
For reading in a browser

The polished, operator-facing layer. When a document needs charts, colour coding, tables, or a visual layout that's easier to scan than plain text, it gets an .html version. These embed the design system's colour tokens so they respect dark mode and match the brand. You open them in a browser ‚Äî they're not meant to be edited directly.

Examples
startup-loop-workflow.user.html startup-loop-files.user.html startup-loop-file-types.user.html

üëÅ Agent writes. Open in browser and read.

.yaml
Schema & configuration
The rules engine

YAML files define the structure that makes the loop work: which stages exist and in what order, what each stage requires before it can start, how deliverables are routed to the right skill, and what vocabulary is valid. The agent reads YAML as authoritative configuration ‚Äî if the stage graph lives in a .yaml file, that's what determines what comes next.

Examples
loop-spec.yaml deliverable-routing.yaml process-assignment-v2.yaml workstream-workflow-taxonomy-v2.yaml

‚öôÔ∏è Loop maintainers write. Agent reads as law.

.ts
TypeScript ¬∑ compiled code
The engine room

TypeScript is what runs under the hood. The startup loop processors (metrics aggregation, manifest updates, stage derivation, growth accounting) are all .ts files. So are the MCP server tools that give the agent access to live shop data, Gmail, analytics, and the BOS API. You never edit these directly ‚Äî they're the compiled machinery.

Examples
derive-state.ts s10-growth-accounting.ts tools/seo.ts tools/loop.ts metrics-aggregate.ts

üîß Engineers write. Runs automatically.

.sh
Shell script ¬∑ automation
The safety layer

Shell scripts are the enforcement layer. They run automatically at key moments ‚Äî before a commit, before a push, during CI ‚Äî and they stop bad things from happening. The writer lock (which prevents two agents from overwriting each other's work), the pre-push safety checks, and the CI validation gates are all .sh files. They say yes or no; they don't negotiate.

Examples
require-writer-lock.sh pre-push.sh validate-changes.sh with-writer-lock.sh telemetry-log.sh

üîí Engineers write. Runs at git hooks & CI.

.json
Data & manifests
Structured data

JSON holds static data, generated indexes, and package configuration. In the startup loop specifically: email templates the MCP server uses, shop inventory and product data, the stage-operator map the agent looks up when routing between stages, and the discovery index that tracks which plans are active. Unlike YAML, JSON is rarely written by hand ‚Äî it's either generated or maintained by tooling.

Examples
email-templates.json stage-operator-map.json products.json inventory.json package.json

How they relate to each other

The loop has three layers. File types cluster by layer.

Layer A ‚Äî Strategy & Standing Knowledge
.user.md ‚Äî strategy docs you fill in
.md ‚Äî agent-written strategy packs
.json ‚Äî shop data & product inventory
Layer B ‚Äî Feature Build Loop
.md ‚Äî fact-find, plan, task artifacts
.user.md ‚Äî build-record & results-review
.html ‚Äî operator dashboards & guides
Infrastructure ‚Äî Loop Machinery
.yaml ‚Äî stage graph, routing rules
.md ‚Äî skill instructions & contracts
.ts ‚Äî runtime processors & MCP tools
.sh ‚Äî safety hooks & CI gates

The naming conventions that matter

Within .md files, the file name carries extra meaning.

<name>.md
Fully agent-produced. No action needed from you. Used as input by the next step in the pipeline.
<name>.user.md
Human input required. Either you wrote it, or you need to confirm it before the loop advances.
YYYY-MM-DD-<name>.user.md
Dated snapshot. One new file per run. History is preserved. Agent reads the most recent.
<BIZ>-<name>.md
Per-business flat file. One file per business, overwritten on each run. Always reflects the latest.
SKILL.md
Skill instruction file. Tells the agent exactly what a skill does, what it's allowed to do, and how to run it.
<name>-r<N>.user.md
Revision suffix. Used when you go back for another round ‚Äî r2, r4, etc. Common in naming sprints.
