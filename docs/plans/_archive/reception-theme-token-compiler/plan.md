# Reception Theme Token Compiler Retrofit Plan

## Summary

Reception is the last major app still using a hand-authored `tokens.css` generated by a separate script. This plan retrofits reception to use the shared `generateThemeCSS()` compiler pipeline via a purpose-written `theme-css-config.ts`. The compiler already supports `tokenVarMap` and `darkSelector`, which are the two key extension points this plan uses. The generated CSS file is a known superset of the current `tokens.css`: it adds two properties (`color-scheme: light/dark` and `--theme-transition-duration: 200ms`) that the compiler always emits but `tokens.css` does not contain. All token vars and values are identical. The plan updates `globals.css` to import the generated file, adjusts `tailwind.config.mjs` shade bridge entries, and adds two parity tests as a CI gate. The parity tests cross-check the generated file against both the compiler output AND the current `tokens.css` to guard against tokenVarMap population errors.

## Goals

- Create `packages/themes/reception/src/theme-css-config.ts` mapping all tokens to the compiler via `tokenVarMap`.
- Generate `apps/reception/src/styles/theme-tokens.generated.css` with all token vars from current `tokens.css` plus two known additive properties (`color-scheme` and `--theme-transition-duration`) that the compiler always emits.
- Add `packages/themes/reception/__tests__/generated-parity.test.ts` and `coverage-parity.test.ts`, both passing in CI.
- Wire the generated file into `apps/reception/src/app/globals.css` replacing the direct `tokens.css` import.
- Update `tailwind.config.mjs` shade bridge to use `var(...)` instead of `hsl(var(...))` for shade entries, since shade vars now contain full `hsl()` values.

## Non-goals

- Do not change any colour values — pipeline retrofit only, no rebrand.
- Do not migrate `globals.css` `--reception-*` runtime aliases into the compiler.
- Do not add or remove shade families.
- Do not change the `ThemeCSSConfig` interface in `@themes/base`.
- Do not touch the existing `tokens.css` file — it stays as-is until TASK-05 swaps the import.

## Constraints & Assumptions

- Constraints:
  - Shade tokens must emit as full `hsl()` strings in the generated file (e.g., `hsl(330 55% 66%)`), not bare triplets — Tailwind v4 cascade rule.
  - Semantic tokens keep bare HSL triplets (consumed via `hsl(var(...))` in Tailwind config).
  - The compiler `darkSelector` supports one selector only. Reception requires two dark blocks: `@media (prefers-color-scheme: dark) :root` and `html.theme-dark`. The generate script must emit both by post-processing the compiler output.
  - The current `tokens.css` `:root` block stores BOTH base vars (e.g., `--color-primary`) AND `-dark`-suffixed siblings (e.g., `--color-primary-dark`). The `tokenVarMap` must include both. Base vars use `dark: "var(--varname-dark)"` to reproduce the dark-mode alias pattern. `-dark` suffix vars are light-only (no `dark` override).
  - 16 runtime alias vars in `globals.css` `:root` stay app-level — excluded from `tokenVarMap`.
  - Tests run in CI only — never locally.
  - Writer lock required before committing.
- Assumptions:
  - `fonts: {}` in `assets.ts` — `fontVarMap` must be empty (`{}`). Font vars (`--font-sans`, `--font-mono`) go into `tokenVarMap` as light-only entries since they have no dark variant.
  - The compiler always emits `color-scheme: light;` in `:root` and `color-scheme: dark;` in the dark block, and always emits `--theme-transition-duration` from `profile.motion.durationNormal`. The current `tokens.css` does NOT contain either of these. The generated file will therefore be a superset — it contains all vars from `tokens.css` plus these two compiler-mandated additions. This is explicitly known and acceptable (additive only; no existing consumers read `--theme-transition-duration` in reception). The parity test must account for this by checking every var in `tokens.css` appears in the generated file — but NOT requiring the generated file to be exactly equal to `tokens.css`.

## Analysis Reference

- Related analysis: `docs/plans/reception-theme-token-compiler/fact-find.md`
- Selected approach inherited:
  - Use `tokenVarMap` code path in the compiler (not `colorVarMap`/`derivedVars`).
  - Compiler `darkSelector: "html.theme-dark"` for the class-toggle dark block; generate script post-processes to also emit `@media (prefers-color-scheme: dark) :root { ... }` block.
  - Shade tokens stored as `hsl()` strings in tokenVarMap light values.
  - `tailwind.config.mjs` shade bridge changed from `hsl(var(...))` to `var(...)`.
- Key reasoning used:
  - `tokenVarMap` was already added to the compiler as a prerequisite — it is the right path for TokenMap-shaped data.
  - Option A (single `darkSelector`) is simpler than multi-selector since the generate script can post-process; compiler interface stays unchanged.
  - The `-dark` suffix siblings must appear in `:root` because downstream consumers (`globals.css` runtime aliases) reference them directly (e.g., `hsl(var(--color-bg-dark))`).

## Selected Approach Summary

- What was chosen:
  - `theme-css-config.ts` with `tokenVarMap` populated from `tokens.ts` — both base vars and `-dark` suffix vars as flat entries; shade light/dark values wrapped in `hsl()`.
  - `darkSelector: "html.theme-dark"` in config.
  - Generate script: call `generateThemeCSS()`, then post-process the output to emit a second `@media (prefers-color-scheme: dark) :root { ... }` block with identical dark var assignments.
  - Include `--theme-transition-duration: 200ms` in generated file (additive, harmless, expected from profile).
- Why planning is not reopening option selection:
  - Compiler already has `tokenVarMap` and `darkSelector` — the prerequisite work is confirmed complete.
  - Option B (keep dark blocks manual in globals.css) would leave a partial migration; the generate-script post-processing approach achieves full migration more cleanly.
