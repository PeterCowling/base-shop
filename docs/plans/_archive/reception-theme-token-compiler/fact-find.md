# Reception Theme Token Compiler Retrofit Fact-Find

## Scope

### Summary

The brikette app now generates its theme CSS from a structured three-layer system (assets → design-profile → ThemeCSSConfig → compiler → generated CSS file). Reception still relies on a hand-authored `tokens.css` generated by a separate `build-tokens.ts` script. This work retrofits reception to use the same `generateThemeCSS` compiler pipeline, with a purpose-written `theme-css-config.ts` as the bridge layer.

The primary complexity is the 35+ bar/POS shade colour families. These use raw HSL triplets (`330 55% 66%`) in `tokens.css` but must ultimately resolve to valid CSS colours in Tailwind v4 utility classes. The Tailwind v4 cascade fix (HSL triplets in unlayered CSS → full `hsl()` values required) applies here. The compiler must emit these shade tokens as `hsl(H S% L%)` strings, not bare triplets.

### Goals

- Create `packages/themes/reception/src/theme-css-config.ts` mapping all reception tokens to the compiler.
- The compiler output (`theme-tokens.generated.css`) must be a byte-for-byte semantic equivalent of the current `tokens.css`.
- Wire the generated file into `apps/reception/src/app/globals.css` replacing the direct `tokens.css` import.
- Add `packages/themes/reception/__tests__/generated-parity.test.ts` and `coverage-parity.test.ts` modelled on brikette equivalents, adapted for reception's token shape.
- Both tests must pass before swapping the import.

### Non-goals

- Do not change any colour values — this is a pipeline retrofit, not a rebrand.
- Do not change `tailwind.config.mjs` colour bridge entries.
- Do not migrate `globals.css` `--reception-*` runtime aliases into the compiler (they are app-level CSS, not theme tokens).
- Do not add or remove shade families.
- Do not change the `ThemeCSSConfig` interface in `@themes/base` (extend via `derivedVars` if needed).

### Constraints & Assumptions

- Constraints:
  - Shade tokens must be emitted as full `hsl()` strings (e.g. `hsl(330 55% 66%)`) in the generated file — not raw triplets — so they survive the Tailwind v4 cascade. See CLAUDE.md "Tailwind v4 CSS Cascade + Shade Token Fix".
  - Semantic tokens (primary, accent, surface, border, etc.) keep raw HSL triplets because they are consumed via `hsl(var(...))` in Tailwind config and in globals.css `receptionSemanticColors`.
  - Dark mode is triply defined: `@media (prefers-color-scheme: dark)`, `html.theme-dark`, and the current compiler only emits `.dark`. The generated file must reproduce all three blocks (media query + class selector) or the globals.css must handle the mapping.
  - The compiler (`generateThemeCSS`) currently emits `:root { }` and `.dark { }` blocks only. Reception uses `@media (prefers-color-scheme: dark) :root { }` and `html.theme-dark { }` patterns, not `.dark { }`.
  - Tests run in CI only — never locally.
  - Writer lock must be acquired before committing.
- Assumptions:
  - The `ThemeCSSConfig` interface's `derivedVars` field is the appropriate extension point for shade tokens, since `colorVarMap` is tied to `assets.brandColors` (which reception does not use for shades).
  - OR: the compiler needs a new `shadeVarMap` extension to emit the `--color-*Shades-row*` family as `hsl()` strings.
  - The generated file name will be `apps/reception/src/styles/theme-tokens.generated.css` (parallel to brikette).

### Dark Mode Architecture

Current `tokens.css` uses three identical dark blocks:

1. `@media (prefers-color-scheme: dark) :root { ... }` — system dark preference
2. `html.theme-dark { ... }` — explicit class toggle (ThemeModeContext)

Both blocks swap each semantic token to its `-dark` sibling (e.g. `--color-primary: var(--color-primary-dark)`), and for shade tokens do the same.

**Critical gap with compiler:** `generateThemeCSS` emits `.dark { ... }`, not `@media (prefers-color-scheme: dark) :root { ... }` or `html.theme-dark { ... }`. Reception's dark-mode class is `theme-dark` on `html`, not `dark`.

Resolution strategy: the generated file should emit a `@media` block and `html.theme-dark` block instead of (or in addition to) `.dark`. Two options:

- **Option A (preferred):** extend the compiler to support a `darkSelectors` config key listing the selectors/at-rules to emit dark vars under.
- **Option B (simpler, no compiler change):** keep the generated file as `:root` light-only block, and let `globals.css` reproduce the dark swap blocks by importing the generated file and continuing to define the media/class blocks manually (but populated by referencing the `-dark` suffix vars from `tokens.ts`).
- **Option C:** post-process the `.dark` block to emit it twice under different selectors.

Option B avoids changing the compiler and keeps the test scope smaller — the generated file covers `:root` vars plus the `-dark` value pairs, and `globals.css` retains the dark swapping logic. But this means partial migration only.

Option A is the cleanest long-term but requires a compiler interface change.

**Recommendation:** Option A — add `darkSelectors?: string[]` to `ThemeCSSConfig`, defaulting to `[".dark"]` for brikette backward compat. Reception specifies `["@media (prefers-color-scheme: dark) :root", "html.theme-dark"]`.

### Shade Token Format Strategy

The Tailwind v4 cascade problem:
- `tokens.css` (unlayered, wins cascade): `--color-pinkShades-row1: 330 55% 66%`
- Tailwind utility sees `background-color: var(--color-pinkShades-row1)` → resolves to `330 55% 66%` → invalid CSS colour → transparent.

Fix confirmed in CLAUDE.md: store shade tokens as `hsl(330 55% 66%)` in the generated CSS, not bare triplets.

In the generated file:
```css
:root {
  --color-pinkShades-row1: hsl(330 55% 66%);
  --color-pinkShades-row1-dark: hsl(330 25% 22%);
  ...
}
```

The `tailwind.config.mjs` `receptionShadeColors` already wraps with `hsl(var(...))`:
```js
"pinkShades-row1": "hsl(var(--color-pinkShades-row1))"
```
So after the fix: `hsl(var(hsl(330 55% 66%)))` — this is **invalid**. We need a different Tailwind approach for shades: since the var now resolves to a full `hsl()` string, the Tailwind config should use the var directly without wrapping:
```js
"pinkShades-row1": "var(--color-pinkShades-row1)"
```

This is the key change to `tailwind.config.mjs` for shade colours — the `hsl(var(...))` wrapper must be dropped for shade entries when the var value is already a full `hsl()` string.

**Semantic tokens remain unchanged** — they keep bare triplets in CSS, and `hsl(var(...))` in the Tailwind config.

### Patterns & Conventions Observed

- `tokens.ts` TokenMap uses `--color-primary: { light: "...", dark: "..." }` shape — clean pairing.
- `tailwind-tokens.ts` flattens this to `{ "--color-primary": "142 72% 30%", "--color-primary-dark": "142 55% 48%", ... }` — useful for alternate consumers.
- The compiler `generateThemeCSS` currently only supports:
  - `colorVarMap` → `assets.brandColors` (not applicable for reception semantic tokens)
  - `fontVarMap` → `assets.fonts` (applicable — `--font-sans`, `--font-mono`)
  - `rgbVarMap` → RGB triplets from hex (not applicable for reception)
  - `derivedVars.light/dark` → arbitrary key-value pairs

This means **all 70 semantic token vars** would need to go into `derivedVars.light` and `derivedVars.dark` unless a new compiler section is introduced. This is viable but verbose. A cleaner approach is a new `tokenVarMap` section (separate from `colorVarMap` which requires `brandColors`) that reads from a raw `TokenMap`.

**Alternative approach for compiler:** rather than using `ThemeCSSConfig` at all, the reception compiler could call a simpler `generateFromTokenMap(tokens: TokenMap): string` function that:
1. Emits `:root` with all `light` values (semantic as bare triplets, shades as `hsl()` strings)
2. Emits dark selector blocks with all `dark` overrides using the same format

This would be a second code path in the compiler, simpler and more direct for the TokenMap shape that reception already has.
